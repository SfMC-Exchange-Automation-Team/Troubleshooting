# Run Stage 1 SOP analysis

Stage 1 analyzes the current Exchange patching SOP and converts known gaps into structured findings. It is a no-change stage: it does not connect to Exchange servers, alter services, enter maintenance mode, or modify patch media.

Use this stage before DAG discovery and pre-check implementation to confirm that the toolbox has enough process information to advance safely.

## Prerequisites

- Windows PowerShell 5.1.
- Local access to the EPO Toolbox project folder.
- A valid `Config\ExchangeCuPatch.config.psd1` file.

Exchange Management Shell, Exchange server connectivity, and CU media are not required for Stage 1.

## Run Stage 1 from the main toolbox

From the project root, run:

```powershell
cd 'C:\Users\cuhaafke\OneDrive - Microsoft\Documents\Microsoft Scout\EPO-Toolbox'
.\EPO-Toolbox.ps1 -Stage Auto -ValidationOnly
```

When `StageAwareness.CurrentStage` is set to `SopAnalysis`, `-Stage Auto` dispatches to `Invoke-ExchangeCuStage1SopAnalysis.ps1`.

You can also run Stage 1 explicitly:

```powershell
.\EPO-Toolbox.ps1 -Stage SopAnalysis -ValidationOnly
```

## Run Stage 1 directly

Use the stage script directly when testing the Stage 1 module without the main dispatcher:

```powershell
.\Invoke-ExchangeCuStage1SopAnalysis.ps1 -ValidationOnly
```

To write output to a specific location, use `-OutputRoot`:

```powershell
.\Invoke-ExchangeCuStage1SopAnalysis.ps1 `
    -ValidationOnly `
    -OutputRoot '\\central-share\ExchangeCU\Runs'
```

## Parameters

| Parameter | Required | Description |
| --- | --- | --- |
| `-ConfigPath` | No | Path to the toolbox configuration file. Defaults to `Config\ExchangeCuPatch.config.psd1`. |
| `-OutputRoot` | No | Root folder for run output. Defaults to `%TEMP%\ExchangeCuDagPatch`. |
| `-CorrelationId` | No | GUID used to correlate run metadata, events, evidence, and summaries. A new GUID is generated by default. |
| `-ValidationOnly` | No | Indicates that this is a validation-only run. Stage 1 is no-change regardless, but the flag is emitted in output. |

## Finding categories

Stage 1 currently evaluates the following SOP areas:

| Phase | Purpose |
| --- | --- |
| `PackageStaging` | Identifies requirements for ISO path, hash validation, extraction validation, and ADS handling. |
| `CredentialAccess` | Confirms credentials remain external and are not persisted by the toolbox. |
| `SplunkHandling` | Requires stateful Splunk forwarder disable/restore behavior instead of startup-type-only changes. |
| `AvReadiness` | Captures CrowdStrike/AV readiness and requires an explicit approved override before any disable action. |
| `MaintenanceMode` | Identifies required DAG maintenance gates such as transport drain, queue redirect, ServerWideOffline, activation policy, and cluster pause validation. |
| `InstallExecution` | Requires absolute `Setup.exe` path use, stdout/stderr capture, setup log tailing, no-log-growth detection, and failure review. |
| `PostInstallValidation` | Converts post-install checks into structured pass/fail gates. |
| `LoadBalancer` | Tracks whether load balancer handling is none, manual, or scripted. |

## Interpret the result

The command returns an object with these key fields:

| Field | Description |
| --- | --- |
| `CorrelationId` | Unique run identifier. |
| `StageAwareness` | Current stage, stage index, total stages, previous stage, next stage, and configured stage order. |
| `Status` | Overall result: `Pass`, `Warning`, or `Blocked`. |
| `Severity` | Highest severity produced by current findings. |
| `Findings` | Per-phase SOP findings. |
| `RequiredInputsForNextStage` | Unique list of information needed before advancing. |
| `NextStage` | The next stage from the configured stage order. |
| `RunPath` | Folder where run evidence was written. |
| `EvidenceFile` | JSON evidence file for the Stage 1 result. |
| `SummaryFile` | CSV summary file for finding review. |

## Current expected behavior

With the current starter configuration, Stage 1 is expected to return a warning/critical status because several production-specific values are placeholders or still require confirmation.

Examples include:

- Final CU ISO path.
- Expected ISO hash.
- Xcopy syntax used by the patching team.
- CrowdStrike exclusion validation method.
- Load balancer handling requirements.
- No-log-growth threshold and approved diagnostic behavior.

This is expected. Stage 1 is designed to surface required process inputs before later stages can safely make changes.
